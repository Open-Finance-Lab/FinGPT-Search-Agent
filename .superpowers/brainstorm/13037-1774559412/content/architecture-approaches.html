Architectural Approaches

How to evolve Agentic FinSearch from search agent → financial output validator

A

Validation Layer on Top (Recommended)

Keep existing search agent intact. Add a new validation pipeline alongside it.

  • Browser extension gains a "Validate" button that works on chatgpt.com, claude.ai, gemini.google.com
  • New backend endpoint: /validate/ — receives raw model output text, returns claim-level verification
  • Reuses existing MCP tools (Yahoo Finance, TradingView, SEC EDGAR) as ground truth oracles
  • New Claim Decomposer module (LLM-powered) breaks response into atomic claims
  • New Claim Verifier module checks each claim against structured data
  • Existing search agent becomes "demo mode" — shows what a validated search experience looks like

Pros

  • Preserves everything you have
  • Fastest path to demo
  • Clear separation of concerns

Cons

  • Two systems to maintain
  • Validation pipeline is new code
B

Validation-First Rebuild

Rebuild around validation as the core. Search agent becomes one consumer of the validation engine.

  • Validation pipeline is the primary backend service
  • Search agent is refactored to pipe its own responses through the validator
  • Single unified architecture — no "old system + new system"
  • Frontend is validation-first: extension always shows accuracy indicators

Pros

  • Cleaner long-term architecture
  • One system, one mental model
  • Stronger narrative for paper

Cons

  • Significant rewrite — weeks of work
  • Risk of breaking working features
  • Delays the demo
C

Standalone Validator (New Project)

Build the validator as a separate service. Agentic FinSearch is just the proof-of-concept context.

  • New standalone backend service focused purely on validation
  • New lightweight browser extension (or separate mode in existing one)
  • Agentic FinSearch stays untouched as the "before" in the narrative
  • Validator service can be independently deployed, scaled, pitched

Pros

  • Zero risk to existing system
  • Clean codebase for the new vision
  • Easiest to pitch independently

Cons

  • Doesn't reuse your MCP infrastructure
  • More total code to write
  • Loses the "evolution" narrative
